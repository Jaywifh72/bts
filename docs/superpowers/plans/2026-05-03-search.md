# Search — Implementation Plan

> Sub-feature of the web app. Adds a single search box that ranks productions, people, gear (manufacturers, series, items), and VFX houses across the entire site.

**Goal:** A `/search?q=...` page returns the most relevant entities across all four browse-able types, served through the existing typed query layer with no external dependencies.

**Architecture:** Postgres `pg_trgm` extension + GIN trigram indexes on the human-readable name columns. Single union query that scores rows with `similarity()` and returns the top N per category.

Why pg_trgm and not Algolia/Meilisearch:
- Corpus is ~1000 entities. pg_trgm handles this in single-digit ms.
- Zero new infra and zero recurring cost.
- Easy to swap for Algolia later by replacing one query function.

**Tech Stack:** Postgres 16 (already deployed), Drizzle, Next.js 14 — no new runtime dependencies.

## Task 2: Search query function

**Files:**
- Create: `packages/db/src/queries/search.ts`
- Modify: `packages/db/src/index.ts` — re-export
- Test: `packages/db/src/tests/search.test.ts` (create)

The single query unions across all entity types with a per-type score and a category tag:

```typescript
export type SearchResult = {
  category: 'production' | 'person' | 'manufacturer' | 'series' | 'item' | 'vfx_house';
  slug: string;
  display: string;          // primary name shown in result list
  subtitle: string | null;  // year, role, manufacturer, etc — context for disambiguation
  href: string;             // ready-to-link path
  score: number;            // 0-1 similarity, higher better
};

export async function search(db, q: string, perCategoryLimit = 10): Promise<SearchResult[]>
```

Implementation: a UNION ALL of seven SELECTs, each computing `similarity(name, ${q}) AS score` and filtering on `name % ${q}` (the % operator uses the trgm index). Outer ORDER BY score DESC then display ASC. The seven SELECTs:
- productions (title + original_title — match either)
- people
- manufacturers
- series (with manufacturer for href)
- items (with manufacturer + series for href)
- vfx_houses

Wrap each in a sub-select that LIMITs to `perCategoryLimit` so one popular category doesn't drown the others. Use `set_limit(0.15)` per session so the `%` threshold is loose enough to catch typos but tight enough to discard noise.

Tests:
- search('dune') returns Dune: Part Two as a production
- search('greig') returns Greig Fraser as a person
- search('alexa 65') returns the ARRI ALEXA 65 series and items
- search('framestre') (typo) still returns Framestore via fuzzy match
- search('') returns []

Commit: `feat(db): add multi-entity trigram search query`

## Task 4: /search results page

**Files:**
- Create: `apps/web/app/search/page.tsx`
- Create: `apps/web/components/search/ResultsByCategory.tsx`

Server component reads `searchParams.q`, calls `search(db, q, 10)`, groups results by category, renders one section per category in fixed order: Productions, Crew, VFX Houses, Manufacturers, Series, Items. Each section shows up to 10 with a "View all" link to the existing index page (`/films`, `/crew`, etc.) when more might exist.

Empty state: if `q` is empty, show a static suggestion list ("Try: Dune, Greig Fraser, ALEXA 65"). If query has no matches, show a polite no-results message.

Page metadata: `noindex` so Google doesn't surface result pages.

Commit: `feat(web): add /search results page`
